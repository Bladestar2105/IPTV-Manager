## 2026-03-02 - SQLite JSON Aggregation for Large Datasets
**Learning:** Returning thousands of individual rows from `better-sqlite3` and iterating over them in Node.js to construct complex nested JSON objects (like the EPG schedule map) is memory intensive and significantly slower than offloading the JSON construction to SQLite. However, when using `json_group_array`, SQLite does not guarantee the order of elements within the array unless the grouping is applied to an already-sorted dataset (using a subquery with `ORDER BY`).
**Action:** Use `json_group_array(json_object(...))` in SQLite queries to aggregate large collections. If order matters (e.g., chronological EPG events), ensure you select from a subquery that has the `ORDER BY` applied before the outer grouping query. Always benchmark optimizations before committing, as SQLite JSON performance vs. Node.js native object creation can vary based on dataset size.
## 2026-03-02 - SQLite json_group_object wrapper for full JSON mapping
**Learning:** While offloading JSON building partially to SQLite via `json_group_array` or `json_object` helps reduce memory overhead vs native object instantiation, returning a large array of objects from SQLite still incurs a heavy `for...of` mapping loop in Node.js where `JSON.parse` is called thousands of times. Wrapping the query in an outer `json_group_object(id, json(value))` allows SQLite to natively construct the final complete JSON dictionary. Returning one massive JSON string from SQLite and calling `JSON.parse()` **once** in Node.js proved to be ~35% faster than the partial approach in parsing large EPG schedules.
**Action:** Whenever constructing a full dictionary or list from database rows (like `schedule[channel_id] = [...]`), consider utilizing an outer `json_group_object` or `json_group_array` query to map the whole result structure inside SQLite, then extracting the single string and parsing it in Node.
## 2026-03-02 - Direct JSON String Transmission from SQLite to Express
**Learning:** When using SQLite's native JSON aggregation functions (like `json_group_object` or `json_group_array`), the result is returned to Node.js as a fully formed JSON string. If this data is intended to be sent directly as an HTTP response, calling `JSON.parse()` on the string only to have Express's `res.json()` immediately call `JSON.stringify()` on it again is highly inefficient. It wastes CPU cycles and unnecessarily spikes Node.js memory usage by building temporary JavaScript objects.
**Action:** When a database query returns a complete JSON string that serves as the final API response, bypass Node.js serialization overhead. Set the HTTP header explicitly (`res.setHeader('Content-Type', 'application/json');`) and send the raw string directly using `res.send(jsonString)`.
## 2026-03-02 - SQLite strftime for XMLTV Date Formatting
**Learning:** When generating massive XMLTV files, iterating through hundreds of thousands of programs and formatting dates natively in Node.js (using `Date.setTime()` and string concatenation) introduces significant V8 memory overhead and slows down the event loop. By offloading date string generation to SQLite's highly optimized C implementation of `strftime('%Y%m%d%H%M%S +0000', column, 'unixepoch')`, we can completely eliminate millions of JS object instantiations and string concatenations.
**Action:** When streaming large exports containing dates (like XMLTV), offload date string formatting to SQLite directly in the `SELECT` query rather than doing it computationally in the Node.js mapping loop.

## 2026-03-08 - Set-based Lookups for Array Filtering
**Learning:** Calling `Array.prototype.includes` inside a `filter` loop over large datasets results in $O(N \times M)$ complexity. For a playlist with 10,000 channels and 5,000 allowed IDs, this results in up to 50 million comparisons.
**Action:** To ensure $O(N + M)$ complexity, always convert the inclusion list into a `Set` before the filter loop. Using `Set.prototype.has` provides $O(1)$ lookup time, which in benchmarks reduced filtering time from ~48ms to ~1.5ms for a 10k/5k dataset.
