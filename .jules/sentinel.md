## 2025-02-14 - Fix Server-Side Request Forgery (SSRF) via Follow Redirects
**Vulnerability:** The application was vulnerable to SSRF because it used raw `fetch` from `node-fetch` with `redirect: 'follow'` to proxy streams and fetch VOD/series info. While the initial provider URLs were validated against unsafe domains/IPs via `isSafeUrl`, `node-fetch` would blindly follow redirects to any URL (including `localhost` or private IPs), bypassing the initial checks.
**Learning:** `node-fetch` handles redirects internally and does not invoke custom DNS lookup agents (like those used for DNS rebinding protection) for plain IP addresses in redirect `Location` headers. Validating the initial URL is insufficient if redirects are followed automatically.
**Prevention:** Always use the custom `fetchSafe` wrapper from `src/utils/network.js` for outbound HTTP requests. It securely handles redirects manually by recursively re-verifying every new `Location` URL against `isSafeUrl` before fetching. Do not pass custom `agent` options to `fetchSafe` as it securely manages its own DNS-safe agents.## 2025-03-01 - Fix Session Invalidation on Password Change
**Vulnerability:** Existing user sessions (like `player_session` and in-memory token caches) remained fully active even after a user (or admin) successfully changed their account password, leaving the user susceptible to active, unauthorized connections maintaining their access.
**Learning:** This occurred because `authController.js` and `userController.js` modified the password hash directly but did not include explicit instructions or API calls to immediately purge corresponding active session state from SQLite and Map caches.
**Prevention:** Whenever sensitive credentials are fundamentally updated or revoked, the session invalidation logic must explicitly clear both persistent database representations (like `temporary_tokens` for player sessions) and any fast-path in-memory object caches (like `tokenCache` in `authService.js`).
